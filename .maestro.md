# Maestro Workflow Context
Generated: 2026-07-19 (teach-maestro interview with Pavel)

## Models & Providers
- Primary: Claude (Fable 5 / Opus tier) through Claude Code (CLI + desktop) as the interactive development agent for this repository. Other assistants (GPT / Gemini / similar) are used occasionally as side tools.
- The application itself makes NO AI/API calls — no model API keys in the codebase, no in-app AI features planned. AI is a development tool here, not a product component.
- Context windows: standard Claude Code sessions (~200K tokens), automatic summarization for long sessions. No special long-context requirements.

## Workflow Architecture
- **Strictly single-agent**: one interactive Claude Code session; subagents (Explore/Plan/parallel task agents) are unwanted.
- Input: a bug report with precise reproduction steps, or a feature request, written in Russian by Pavel (sole developer/operator, technical, makes his own architectural decisions).
- Output: working implementation committed to git, pushed to BOTH remotes (origin = GitLab + github), and deployed to production (`/srv/songs`, deploy = `git pull` over ssh). Standing permission to commit/push without confirmation.
- Agent tooling in use: file read/edit + search, PowerShell/Bash (terser minification, `php -l` against PHP 7.2), git, remote production MySQL (server.winsys.lv:43306) for live diagnostics, ssh deploy (occasionally blocked by the permission classifier — then Pavel runs the command himself), persistent cross-session project memory + CLAUDE.md conventions. Browser-automation tools are not part of the workflow.
- Mechanical contract after any JS edit: `npx terser public/js/foo.js -o public/js/foo.min.js --compress` (never `--mangle`) + bump `?v=N` in every template referencing the file.
- Mechanical contract for any UI change: every new or changed user-facing string goes through the UI i18n system — keys added to all four dictionaries (ru/de/en/lt), rendered via `window.t('key')` in JS / `T::s('key')` in PHP; hardcoded UI text is not acceptable.

## Quality & Evaluation
- Evaluation is manual, on production: Pavel verifies against exact reproduction steps on real pages/devices after deploy; iterative diagnosis continues until the bug is actually gone — "do not declare victory prematurely" is a project rule.
- No automated tests exist today and building a suite is not a current priority; tests are not rejected on principle — working tests would be kept, and the CLAUDE.md §16 stack (PHPUnit → Jest → PHPStan/ESLint → Playwright) remains the reference plan if/when testing starts. No golden examples.
- Most common failure modes:
  1. **Regressions** — a change in one feature silently breaks an adjacent scenario (e.g. server-side display-target enforcement breaking the tech console's song following, July 2026);
  2. **Context loss between sessions** — decisions and agreements have to be re-explained.
- Regression gate: `docs/deploy-checklist.md` — shared-mechanism impact map (consult at change time) + 5-minute production smoke protocol (run after deploying anything that touches a shared mechanism).

## Constraints
- Cost: Claude Code subscription, no hard per-request/per-day budget; token economy is not a driving concern.
- Latency: interactive dialogue pace; correctness beats speed. Long unattended background runs are not part of the workflow.
- Compliance: no formal compliance requirements, no sensitive-data regime (per Pavel). Standing hygiene still applies: secrets never committed (config.php is git-ignored), production data dumps never added to the repo.
- Technical constraints shaping all work: PHP 7.2 on production, MySQL 5.7 (utf8/utf8mb4 mix — `current` table is utf8), AngularJS 1.6.6 + Bootstrap 3 + jQuery, Workerman WebSocket server, **minimal-dependency policy** (no new frameworks or libraries unless essential).

## Priorities
1. **Quality** (highest)
2. Speed
3. Safety
4. Cost (lowest)

Single biggest workflow problem to solve: **regressions**. With no test net existing or planned, impact-checking must happen at change time — before shipping a change to a shared mechanism, trace every consumer of that mechanism (e.g. anything reading/writing the `current` table, WebSocket message consumers, display-target resolution paths) and verify each surviving scenario.
